# Progress Log

Development progress for each implementation phase. Tracks what was built, not release notes.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).

## [Unreleased]

### Added

- (2026-04-28) Updated user-facing documentation for the canonical tracer name feature. The `tracerName` field is now documented in the Configuration Reference table in README with its default derivation logic (Weaver registry manifest `name`, underscores to hyphens) and its known limitation (variable-based `trace.getTracer()` calls are not validated). `docs/rules-reference.md` now lists CDQ-011 in the Tier 2 blocking Code Quality table, removes CDQ-008 from the active run-level rules table (replaced by the new blocking check), and notes CDQ-008 as deleted in the structural changes changelog.

- (2026-04-28) Added CDQ-011 (Canonical Tracer Name), a new blocking per-file validation rule that verifies `trace.getTracer()` string literals match the project's canonical tracer name exactly. The rule is registered in the JavaScript and TypeScript validation chains and runs in the fix loop after each instrumentation attempt. If the agent uses the wrong name (e.g., underscored variant instead of hyphenated), the file fails validation and the fix loop corrects it. When no canonical name was resolved at coordinator level, the rule degrades gracefully (passes) rather than blocking instrumentation. Also added `canonicalTracerName` to `ValidationConfig` so it flows through the full validation chain from `buildValidationConfig` to the rule's `check()` method.

- (2026-04-28) Wired the canonical tracer name into every per-file instrumentation call. The coordinator now resolves the canonical name (via `resolveCanonicalTracerName`) once before the file dispatch loop and passes it down through the options chain — dispatch → `instrumentWithRetry` → `instrumentFile` → `buildSystemPrompt`. The system prompt includes the canonical name in both the Tracer Acquisition section (telling the agent exactly what string to use in `trace.getTracer()`) and the Code Quality section (flagging that it will be validated by a blocking check). Fix-loop feedback prompts also carry a reminder so the agent doesn't revert to a different name on retry attempts. This eliminates the cross-file tracer name inconsistency that CDQ-008 detected but could not fix.

- (2026-04-28) Added `tracerName` field to `orb.yaml` config and a `resolveCanonicalTracerName` function in the coordinator. When `tracerName` is set, it is used as-is. When absent, the canonical name is derived from the Weaver registry manifest's `name` field with underscores normalized to hyphens (e.g., `commit_story` → `commit-story`). This is the foundation for ensuring every per-file instrumentation prompt receives the same tracer name, eliminating inconsistent `trace.getTracer()` calls across multi-file projects.

- (2026-04-28) Removed CDQ-008 (Tracer Naming consistency) — a cross-file advisory rule that detected inconsistent `trace.getTracer()` names across instrumented files but could not fix them, leaving users with a problem they did not cause. The rule lived at coordinator level after all files were already instrumented, making per-file correction impossible. The replacement approach (canonical tracer name injected into every per-file prompt before instrumentation starts, plus a per-file blocking check) is implemented in subsequent milestones of this work. All CDQ-008 references removed from the agent prompt, rule registry, validator, coordinator, language providers, and test suite.

- (2026-04-29) Fixed NDS-003 (Code Preserved) false positives on Prettier multi-line brace style. When an agent formats span lifecycle code with `catch` and `finally` on their own lines (rather than `} catch` and `} finally` combined), NDS-003 flagged them as non-instrumentation additions. Added standalone `catch (...) {` and `finally {` patterns to `INSTRUMENTATION_PATTERNS`, and made the trailing semicolon optional on the existing `throw` pattern to cover `throw error` without `;`. Confirmed from taze eval run-7.

- (2026-04-29) Fixed three false-positive NDS-001 (Syntax Valid) and NDS-003 (Code Preserved) failure classes surfaced by taze eval run-6. First: `checkSyntax()` no longer propagates `lib` from tsconfig — passing `--lib ESNext` (from taze's tsconfig) stripped DOM globals like `console`, causing false TS2584. Without an explicit `--lib` flag, tsc uses the default lib for the target (ESNext default includes DOM). Second: `checkSyntax()` now auto-detects `@types/node` by checking whether `node_modules/@types/node/` exists under the project root; if found and not already in the `types` list, it is added. This mirrors TypeScript's full-project auto-discovery and fixes TS2591 failures on `node:*` imports in projects (like taze) that have `@types/node` installed without an explicit `"types": ["node"]` in tsconfig. Third: NDS-003 now allows compound AND null guards (`if (a != null && b.c !== undefined) {`) in addition to single-condition guards. In run-6, the agent needed to guard two levels of optional access before `span.setAttribute`, creating a catch-22 where omitting the guard caused TS2345 (NDS-001) and including it caused NDS-003.

- (2026-04-29) Fixed two more false-positive NDS-001 (Syntax Valid) failure classes surfaced by taze eval run-5. First: `checkSyntax()` now reads `target`, `lib`, and `types` from the project tsconfig (following `extends` one level) and passes them as `--target`, `--lib`, and `--types` to tsc. Previously the hardcoded `--target ES2022` caused TS2550 on projects using ESNext APIs (e.g. `Array.fromAsync` in taze), and missing `--types node` caused TS2591 on files using `node:*` protocol imports. Second: NDS-003 (Code Preserved) now reconciles `const <name> = <expr>` captures that exist solely to provide a value to `span.setAttribute()`. Previously the capture line appeared in `addedLines` as a non-instrumentation addition; a new `reconcileSetAttributeCaptures()` function clears it when the variable appears in the full instrumented output exactly twice — once in the capture and once in a `span.setAttribute()` call — with no other uses.

- (2026-04-28) Fixed two `checkSyntax()` issues surfaced by taze eval run-4. First: tsc 6.x (bundled in projects like taze) now emits TS5112 when individual files are passed on the CLI alongside a `tsconfig.json` — this was a new enforcement of existing behavior that caused every taze file to fail NDS-001 before instrumentation. Fix: add `--ignoreConfig` to the tsc invocation when the bundled tsc is version 6+ (detected via `getTscMajorVersion()`; tsc 5.x does not support this flag). Second: tsc writes diagnostics to stdout rather than stderr in both tsc 5 and tsc 6, so the NDS-001 error message always showed an empty double-space gap (`${stderr.trim()}` was always empty). Fix: capture stdout alongside stderr and combine both in the error message. Both findings were verified empirically before fixing.

- (2026-04-28) Added agent thinking blocks to `--verbose` output for failed files. Previously `thinkingBlocksByAttempt` was only accessible via the acceptance gate test harness, making it a diagnostic blind spot during real CLI eval runs. Now when `--verbose` is set and a file fails, any thinking blocks from each attempt appear in an "Agent thinking" section after the validation failures section.

- (2026-04-28) Fixed `checkSyntax()` in the TypeScript provider to use the target project's `moduleResolution` and `module` settings instead of hardcoding `--moduleResolution NodeNext`. Previously, passing a file path directly to `tsc` with inline flags bypassed `tsconfig.json` entirely — any project using `"moduleResolution": "Bundler"` (taze, Vite-based tools, esbuild-based tools) failed NDS-001 on every file, including completely unmodified source. The fix walks up from the file's directory to find `tsconfig.json`, reads its `module` and `moduleResolution` values (following `extends` one level), and substitutes those into the per-flag invocation. All other flags (`--strict`, `--skipLibCheck`, etc.) remain hardcoded so project-specific constraints like `verbatimModuleSyntax` are not inherited and don't produce false positives on instrumented output. A new `findTsconfig()` export enables unit testing of the walk-up logic.

- (2026-04-28) Implemented CDQ-005 (startActiveSpan Preferred) as a new advisory validation rule. The rule detects `tracer.startSpan()` calls in agent-generated instrumentation and asks the agent to confirm the choice is intentional. `startActiveSpan()` is preferred because it automatically sets the span as active in context so child operations are correctly parented; `startSpan()` is accepted only when the agent explains which of four legitimate scenarios applies. CDQ-005 replaces the prior CDQ-005 stub ("Async Context Maintained"), which was never implemented, with a fully functional, tested advisory rule. The rule is now registered in both the JavaScript and TypeScript providers and fires in the fix loop.

- (2026-04-28) Fixed COV-005 (Domain Attributes) to correctly track `setAttribute` calls on spans with non-standard variable names. Previously, `extractSetAttributeName` filtered receivers by a `/span/i` regex, so spans assigned to variables like `op` or `s` had their attribute calls silently missed — producing false positives where required attributes appeared absent. The fix uses structural tracking: for `startActiveSpan` the callback's first parameter name is collected, and for `startSpan` the bound variable name is collected. These known names are passed alongside the regex so any standard or non-standard span variable is recognised correctly.

- (2026-04-28) Improved check-failure message quality across the fix loop. Three degraded-operation messages that lacked actionable recovery guidance were updated: (1) the live-check partial warning now states the partial state is advisory and the run completed successfully; (2) the `End-of-run live-check failed (degraded)` message now includes a recovery action directing the user to re-run spiny-orb or check network connectivity; (3) the `SCH-005 (No Duplicate Span Definitions) check failed (degraded)` message now includes an equivalent recovery action. Unit tests were added for Weaver shutdown failure, live-check partial, and live-check unexpected failure message formats.

- (2026-04-27) Added `lastErrorByAttempt` to the diagnostic data captured after each instrumentation attempt. Previously the fix loop recorded only a per-attempt count summary (`errorProgression`) and the full error text from the final attempt (`lastError`). When the agent oscillated — producing different failures on attempt 1 vs attempt 2 vs attempt 3 — there was no way to see how the errors changed across attempts without adding debug logging. The new field stores the complete blocking-failure text from each attempt at the same index as `thinkingBlocksByAttempt`, so `lastErrorByAttempt[0]` is the full text for attempt 1, `lastErrorByAttempt[1]` for attempt 2, etc. The acceptance gate `dumpDiagnostics` function now logs this field alongside `lastError`.

- (2026-04-27) Added a "Validation failures" section to `spiny-orb instrument --verbose` output for failed files. Previously the verbose output for failures showed the failure reason in the status line and agent notes below, but not the full validator error text. The eval team had to dig through agent notes to find the actual rule violation messages — e.g., reconstructing that NDS-001 fired with `TS2352: Conversion of type 'unknown' to type 'Error'` from indirect evidence. Now the full `lastError` text (with complete rule ID and message for every blocking failure from the last attempt) appears in a dedicated "Validation failures (last attempt)" section, flowing directly into `spiny-orb-output.log` when the command is piped through `tee`.

- (2026-04-27) Added `--debug-dump-dir <path>` option to `spiny-orb instrument`. When set, the command writes each file's last instrumented code (the code the agent produced before the validation failure triggered a restore) to the specified directory during the run. This is the CLI equivalent of the acceptance gate test harness's `/tmp/spiny-orb-debug-{label}` file — it lets the eval team inspect what the agent actually produced without modifying the source code or setting up a test harness.

### Changed

- (2026-04-27) Completed the pre-instrumentation analysis pass feature (cross-file manifest, rules reference, README, and diagnostic infrastructure) — all planned milestones for this feature are now done.

- (2026-04-27) Added cross-file instrumentation awareness to the pre-scan pass. When the coordinator instruments a codebase file-by-file, the agent working on each subsequent file now knows which functions from earlier files have already received spans. Previously, if file B called an async function imported from file A, the agent working on B had no way to know whether that imported function was already instrumented — it could either add a redundant span, skip it, or make a confused attempt. Now the coordinator builds a manifest of instrumented function names as it completes each file, and that manifest reaches the pre-scan before the LLM call. The user message gains a directive like "Already instrumented in `./services/orders.js`: `handleOrder`, `processPayment`. Do not re-instrument these." — turning a non-deterministic inference problem into an explicit fact.

- (2026-04-27) Documented the pre-instrumentation analysis pass in the rules reference and README. Each validation rule that has a pre-scan counterpart (COV-001, COV-002, COV-004, RST-001, RST-004, RST-006) now has a "Pre-scan:" note in `docs/rules-reference.md` explaining what the deterministic AST pass computes before the LLM call and what directive it injects. CDQ-007 now documents its import constraint: advisory attribute transformations that require a utility function (e.g., `basename` for path attributes) only apply when that utility is already imported — the agent will not add new non-OTel imports for advisory compliance. The README's "What is this?" section now mentions the pre-scan pass in step 1 of the how-it-works description.

- (2026-04-27) Tightened the CDQ-007 advisory rule to prevent the agent from adding new non-OTel imports when applying attribute transformations. The advisory recommends converting raw filesystem path variables to `path.basename()` values, but the agent was adding `import { basename } from 'node:path'` to files that didn't already import it. NDS-003 correctly blocked those imports as non-instrumentation additions — but the block caused the agent to retry indefinitely. The fix adds an explicit constraint to the CDQ-007 rule text: transformations that require a utility function only apply when that utility is already present in the file; if not, use the raw value and document it as a known limitation in `notes`.

- (2026-04-27) Strengthened the NDS-003 (code preservation) gate rule to make explicit that valid instrumentation never requires removing original code. If an instrumentation approach requires removing something, the approach is wrong — not the code. This addresses a pattern where the agent rationalized removing graceful-degradation try/catch blocks as "simplification." Also extended the NDS-005 (control flow preservation) error message to include a preview of the original try/catch block when it is reported missing. Previously the message only said "Original try/catch block (line 489) is missing," leaving the agent to guess what it removed; now it includes the first 8 lines of the removed block so the agent has specific content to restore. Both changes came out of diagnosing the `index.js` acceptance gate failure using all five diagnostic dimensions (history, actual code, error messages, agent notes, agent thinking) — which revealed the agent was NOT removing the try/catch as assumed from error codes; it was wrapping an inner async call in a sub-span, which changed the block's fingerprint and caused a matching failure.

- (2026-04-27) Added the instrumentation agent's self-reported `notes` field to the acceptance gate diagnostic output. The notes were always populated but never printed, which led to incorrect diagnoses based on error codes alone. The notes revealed that for the `index.js` failure, the agent believed it was correctly preserving the try/catch structure — the NDS-005 fire was a fingerprint mismatch from wrapping `triggerAutoSummaries` with a sub-span, not a removal.

- (2026-04-27) Updated the architecture diagrams to show the pre-instrumentation analysis step. `docs/diagrams/per-file-sequence.mmd` now includes a "Pre-instrumentation analysis (deterministic AST scan)" node between the file-load step and the AI agent, reflecting the new deterministic pass added in M1–M3. `docs/diagrams/orchestrator-overview.mmd` updates the edge label from the orchestrator to the agent to include "pre-scan findings," documenting that the agent now receives structured, function-specific directives alongside the file contents and resolved schema.

### Fixed

- (2026-04-27) Removed line numbers from the NDS-005 "missing block" error message. The message previously said "Original try/catch block (line 489) is missing," which implied the validator matched by line position. It doesn't — it matches by block content. The block preview already shows exactly what was removed; the line number was redundant and led the agent to misdiagnose the failure as a line-shifting issue rather than a content mismatch.

- (2026-04-27) Fixed the NDS-005 (control flow preservation) body anchor to use only the first line of a statement's text, not an 80-character slice of the full multi-line text. The bug: when a multi-line function call like `const result = await doWork('.', {\n  onProgress: ...\n})` is wrapped inside a `startActiveSpan` callback, the extra nesting level adds spaces to continuation lines. The 80-char slice crossed the first newline and captured that indentation, making the original and instrumented anchors differ even though the try/catch was structurally identical. The primary match failed, the fallback match also failed (same anchor used as the non-throwing-catch guard), and NDS-005 reported the block as missing. The fix uses `text.split('\n')[0].slice(0, 80)`, which is stable regardless of nesting depth. This was the root cause of the `index.js` acceptance gate failure that persisted through five rounds of fixes — all prior fixes were correct; the validator was misidentifying preserved code as removed.

- (2026-04-27) Fixed the acceptance gate debug file capture to write the agent's actual instrumented output rather than the restored original. `dumpDiagnostics` was reading from the temp file path, but the retry loop restores that file to the original source after every validation failure (before moving to the next attempt). On a failed result, the temp file always contained the original — making the debug dump useless for diagnosing what the agent actually generated. The fix adds `lastInstrumentedCode?: string` to `FileResult`, populated from the last attempt's `output.instrumentedCode` in `buildFailedResult`. The debug capture now reads from this field instead of disk.

- (2026-04-27) Fixed a contradiction in the pre-scan where async functions with direct `process.exit()` calls (like CLI subcommand handlers) appeared in both the "needs spans" COV-004 list and the "skip this" RST-004 list simultaneously. The pre-scan now applies the same exception as the prompt rule: functions that call `process.exit()` directly don't get a span directive, because `process.exit()` bypasses the span's `finally` block at runtime. This eliminated a case where the agent received conflicting guidance and resolved the conflict by over-instrumenting, causing the try/catch in the calling function to be dropped.

- (2026-04-27) Fixed a class of Anthropic API parse errors being treated as permanent instrumentation failures. When the API returns truncated or malformed JSON due to network issues or streaming interruptions, the fix loop was consuming a retry slot without attempting any code fix. These errors are now classified as transient and retried, consistent with the existing handling for null `parsed_output` responses.

- (2026-04-27) Fixed process.exit() detection in the pre-instrumentation analysis for variable-assigned async arrow functions (e.g., `export const handleSummarize = async () => {}`). The node map used to call `hasDirectProcessExit` was built only from `FunctionDeclaration` nodes; arrow functions assigned to variables were classified as entry points but silently skipped for the process.exit() constraint check. The map now covers both declaration forms, so the minimal-wrapper directive is correctly injected for both function styles.

- (2026-04-27) Fixed a gap in the "all sync exports" early-return heuristic in `instrument-file.ts`. Files that export only synchronous helpers but also contain an unexported `async main()` function — a common CLI pattern — were previously returned early without calling the LLM, which meant `main()` never received a root span. The pre-instrumentation analysis now runs before the early-return checks, and the heuristic is blocked when the pre-scan identifies entry points that need instrumentation. Added a regression test covering this case.

### Added

- (2026-04-27) Added a five-dimension debugging protocol to the acceptance gate test suite. When `index.js` instrumentation failed, the diagnostic output showed only "1 blocking error (NDS-005:1)" — a count with no specifics. The new infrastructure adds: (1) the agent's actual last instrumented output written to `/tmp/spiny-orb-debug-{label}` before the temp dir is cleaned up; (2) `lastError` in the diagnostic JSON, which contains the full validator message including NDS-005's 8-line block preview showing exactly which try/catch was flagged; (3) per-attempt thinking blocks in `thinkingBlocksByAttempt` (up to 2000 chars each), so oscillation failures show the agent's reasoning across all attempts, not just the first. The protocol is documented in `.claude/CLAUDE.md` with specific field names and file paths so future sessions can apply all five dimensions systematically before diagnosing or making code changes.

- (2026-04-27) Extended the pre-instrumentation analysis pass with local import resolution. For each async entry-point function, the pre-scan now walks the function's call expressions and categorizes each identifier-callee call as either locally defined in the same file or imported from another module. The LLM user message gets a per-entry-point breakdown: "In `main()`, async sub-operations — local: `localHelper`; imported (handled elsewhere): `handleSummarize` from `./handlers.js`." This resolves the remaining ambiguity in the "instrument sub-operations instead" guidance: when all sub-operations are imports, the agent knows they belong to their source files and that this file's entry-point span is the coordination point.

- (2026-04-27) Extended the pre-instrumentation analysis pass with four additional rule categories: COV-004 (async non-entry-point functions that also need spans), RST-001 (pure sync functions to skip), RST-004 (unexported functions to skip), and COV-002 (outbound HTTP/DB/messaging call sites in async function bodies). Each category injects concrete function-specific directives into the LLM user message, giving the agent a complete picture of what to instrument and what to leave alone before it reads a single line of source. Also added an early-exit condition: when the pre-scan determines a file has no async functions at all — re-export barrel files and all-sync utility modules — the LLM call is skipped entirely, eliminating a class of NDS-001 oscillation where the agent would correctly decide not to instrument a file but the validation pipeline would still run `tsc --noEmit`, causing spurious failures.

- (2026-04-27) Added a deterministic pre-instrumentation analysis pass that runs before each LLM call. Before asking the agent to instrument a file, the pipeline now uses the same AST infrastructure as the validation rules to compute facts about the source — which functions are entry points requiring spans (COV-001), which of those entry points have direct `process.exit()` calls that interact with span lifecycle (RST-006 vs COV-001 tiebreaker) — and injects those findings as concrete, function-specific directives into the user message. The immediate target was the `index.js` acceptance gate failure, where `main()` and `handleSummarize()` (both async entry points with direct `process.exit()` calls) caused the agent to oscillate between NDS-003/NDS-005 violations and removing all spans entirely. Also tightened the three conflicting prompt rules (COV-001, RST-006, CDQ-001) to resolve the contradiction: COV-001 now explicitly wins over RST-006 for async entry points, with a minimal-wrapper directive that prevents the NDS-005 violation (no `span.end()` before `process.exit()`). New `PreScanResult` type in the `LanguageProvider` interface lets future Python and Go providers add their own analysis without any interface changes.

Updated `docs/ROADMAP.md` to remove the completed multi-language architecture entry and mark Python (#373), Go (#374), and SCH rebuild (#508) as unblocked.

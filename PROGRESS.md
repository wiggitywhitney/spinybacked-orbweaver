### Changed

- (2026-04-22) Bumped `@anthropic-ai/sdk` from `^0.78.0` to `^0.90.0`. CHANGELOG reviewed across the 12 versions: no breaking changes affect spiny-orb's call sites (`messages.stream()`, `zodOutputFormat`, `output_config` structured output parameter, or extended thinking). All 2154 tests and typecheck pass at the new version.

- (2026-04-22) Extended RST-006 (no agent-added spans on `process.exit()` functions) to use container-aware function keys in both the pre-existing-span detection pass (`getFunctionsWithSpans`) and the main traversal (`checkProcessExitSpan`). Class methods now key by `ClassName.methodName` instead of bare method name, preventing false negatives when two classes share a method name and one already has a span. CJS export patterns — `module.exports.foo = async function(){}` and `module.exports = { foo: async () => {} }` — are now detected and properly keyed; previously they produced `fnName = undefined` and were silently skipped. Detection for `FunctionDeclaration` and variable-assigned functions is unchanged.

- (2026-04-22) Updated the rules reference documentation (`docs/rules-reference.md`) with two changes from this work: COV-004 now documents both exemptions (RST-001 utility functions and direct `process.exit()` calls), and RST-006 has its own entry explaining the diff-based detection, the runtime span-leakage mechanism, and the catch-block exception. The smoke test confirmed the fix is working: the agent instrumented `gatherData` and `saveResult` correctly, skipped `main()` without a COV-004 finding, and succeeded in one attempt.

- (2026-04-22) Added a targeted smoke test script (`scripts/smoke-test-process-exit.ts`) that validates the process.exit() fix with a single real API call in 3-5 minutes, versus the ~55-minute full acceptance gate suite. The accompanying fixture (`test/fixtures/smoke/process-exit-instrumentation.js`) mirrors the exact failure pattern: a `main()` function with a top-level `process.exit()` call and an inner try/catch, plus two async sub-operations (`gatherData`, `saveResult`) that are the correct instrumentation targets. The script exits 0 only when: instrumentation succeeds, at least 2 spans are added, and COV-004 does not fire for `main()`.

- (2026-04-22) Updated the agent prompt to teach the LLM about the `process.exit()` instrumentation constraint. COV-004's description now has a clearly marked exception sentence so the agent knows not to span `process.exit()` functions even when they are exported async functions. RST-006 is added to the scoring checklist with the correct catch-block exception: functions where `process.exit()` appears only in a catch block are not exempt from COV-004 and should still be spanned.

- (2026-04-22) Added RST-006, a new advisory validation rule that fires when the instrumentation agent adds a `startActiveSpan` wrapper to a function that calls `process.exit()` directly in its body. This is a diff-based safety net: it only fires on spans the agent added (not pre-existing spans), and it does not fire when `process.exit()` appears only in catch blocks or nested callbacks. Together with the COV-004 exemption, this closes the gap where the agent was being directed to instrument an uninstrumentable `main()` function.

- (2026-04-22) COV-004 (async function span coverage check) now exempts async functions that call `process.exit()` directly in their top-level body. These functions cannot be safely spanned — `process.exit()` bypasses `finally` blocks, which would cause the span to leak and never export. The exemption is intentionally narrow: functions where `process.exit()` appears only inside catch blocks or nested callbacks are NOT exempt, because the happy path can still be safely spanned. This fix eliminates the root cause of the `index.js` acceptance gate failure, where COV-004 was directing the agent to wrap `main()` in a span, causing a cascading NDS-005 violation.

- (2026-04-22) Starting work on COV-004/RST-006 rule changes: COV-004 (async function span coverage) will gain a narrow exemption for functions that call `process.exit()` directly in their body — those functions cannot be safely spanned because `process.exit()` bypasses `finally` blocks. A new advisory rule RST-006 will catch any agent-added spans on such functions as a safety net. Audit confirmed COV-004 has no such exemption today and no existing rule covers this gap.

- (2026-04-21) Added an advisory-only polish pass to the fix loop: when a file passes blocking validation but still has advisory findings (non-blocking quality issues like unnecessary spans or attribute data quality problems), the agent now gets one additional attempt to clean them up. The pass uses the already-passing instrumented code as its starting point — not the original source — so the agent only needs to address the advisory issues rather than re-doing all the instrumentation from scratch. - (2026-04-20) Audited the four API advisory validation rules (API-001 through API-004). Two rules were promoted to blocking after a key insight: both rules check the *instrumented output* for forbidden imports, but a pre-existing forbidden import in the developer's original code is the developer's concern, not the agent's. Narrowing detection to only agent-added imports (by diffing the original file against the instrumented output) makes blocking safe — it only fires when the agent did something wrong, never when the developer had a pre-existing import. API-001 (non-`@opentelemetry/api` OTel package imports) and the import-level API-004 (`@opentelemetry/core` — SDK-internal plumbing) are now blocking with this diff-based approach. API-003 (vendor-specific SDK imports like `dd-trace`, New Relic) was deleted: with diff-based detection it would never fire, and API-001 already covers the conceptual ground. API-002 (checks `package.json` to verify `@opentelemetry/api` is in the right dependency bucket for libraries vs apps) stays advisory — the agent can't modify `package.json`, so blocking would make spiny-orb permanently non-functional for any project with a misconfigured `package.json`. Also activated a dead-code check in API-002 that had been implemented but never wired up: it now also flags library projects that have `@opentelemetry/sdk-*` packages in their dependencies (deployers choose the SDK; libraries shouldn't bundle it), with a message that explicitly attributes this as an OTel project-level recommendation rather than an agent error. The advisory rules audit document was also significantly improved: all downstream action items now have concrete done criteria, negative constraints (what not to do), and scope boundaries so future AI sessions can act on them without re-reading rule implementations.

- (2026-04-20) Audited the SCH schema validation rules. Found two structural problems that require a downstream rebuild: the agent can declare new span and attribute extensions that silently duplicate existing registry entries (no duplicate check runs at declaration time), and SCH-004's post-hoc semantic duplicate detection for attributes runs after the fact when it should catch duplicates the moment the agent declares them. The SCH-001 naming quality fallback (used when the registry has no span definitions) was also found to use an LLM judge to check dotted notation and vagueness — deterministic checks that a regex handles more reliably. The "sparse registry" advisory downgrade for both SCH-001 and SCH-002 turns out to be a workaround for the missing declaration-time duplicate gate rather than a principled design choice. Also discovered that two diverged copies of the SCH rules exist — one in `tier2/` (where acceptance gate tests point) and one in `javascript/rules/` (what runs in production) — with the `javascript/rules/` version being significantly more sophisticated. This duplication must be resolved as part of the rebuild, which is also blocked by a broader question: as Python and Go language providers are added, how should rules that are language-agnostic in concept (span naming, attribute key conformance) share their matching logic across providers? All findings documented in the advisory rules audit document with prescriptive rebuild narratives. SCH-004 is marked for deletion once its type-inference and pre-filtering patterns are migrated to SCH-002's extension acceptance path.

- (2026-04-18) Established that every advisory rule audit must check OpenTelemetry specification alignment before evaluating detection logic. - (2026-04-18) Refactored CDQ-007's filesystem path detection to distinguish two cases that were previously conflated. The check now exempts OTel `file.*` attribute keys and only flags path-like values on non-file attributes (e.g., `span.setAttribute('config.source', filePath)`), where the privacy concern is real.

- (2026-04-18) Promoted three NDS validation rules from advisory to blocking and added a fourth new rule. NDS-004 (exported function signature preservation) and NDS-006 (module system preservation) were advisory as a conservative default — both have direct AST-based detection with no realistic false-positive path, so blocking is the right setting. NDS-005 (control flow preservation — catches removed catch clauses, modified throws, deleted try/catch blocks) had an LLM judge pass that was intended to clear false positives, but no concrete false-positive scenario could be identified that the deterministic matching didn't already handle; the judge was removed and the rule promoted to blocking. NDS-007 is a new blocking rule (replacing the eval-only NDS-005b check): it fires when the agent adds `recordException()`/`setStatus(ERROR)` to a catch block that originally swallowed the error gracefully — returning a default value or empty result rather than rethrowing. Adding error recording to these blocks creates false span errors when the application actually handled the condition successfully. All 2,114 tests pass.

- (2026-04-15) Added a `span_kind` pre-filter to the SCH-005 span deduplication check. Two spans with different `span_kind` values (e.g. CLIENT vs SERVER) represent structurally distinct operation roles and can never be semantic duplicates — the instrumentation on one side of a call is not a duplicate of the instrumentation on the other. This deterministic check now runs before the LLM judge, so the judge only sees structurally compatible pairs. The `SpanDefinition` type now carries `span_kind` alongside `id` and `brief`.

- (2026-04-15) Simplified SCH-005 span deduplication to judge-only detection. The Jaccard similarity script tier was removed — it was cargo-culted from SCH-004 but didn't fit span IDs, which are semantically rich short strings where "are these the same operation?" is a semantic question, not a structural one. The Jaccard pre-screening step also inflated similarity scores because the D-1 namespace filter guaranteed shared prefix tokens. Now all same-namespace span pairs go directly to the LLM judge. When no judge client is available, the check degrades gracefully (returns pass, no findings). All 2,097 tests pass.

# Advisory Rules Audit

**Date**: 2026-04-15 (audit began 2026-04-18)
**PRD**: [#483](https://github.com/wiggitywhitney/spinybacked-orbweaver/issues/483)
**Scope**: All advisory (non-blocking) validation rules, cross-file rules, and orphaned implementations.

**Decision vocabulary**:
- `keep (advisory)` — detection sound, finding actionable; stays advisory (non-blocking, directed in fix loop once the mechanism PRD ships)
- `promote to blocking` — detection sound, fix deterministic and safe; should gate file success
- `refactor` — detection partially sound; scope-narrowed in this PRD
- `rebuild` — wrong algorithm; replacement approach documented below; downstream PRD required
- `delete` — not producing useful signal

### Downstream PRD candidates

### Cross-cutting concerns

- **Human-facing advisory output**: Per-rule `message` fields are written for agent consumption — terse and directive. When advisory findings are surfaced to humans (e.g., an instrumentation report), they will lack the context a human needs: what was detected, why it matters, and what to do. This affects all advisory rules. Requires a design decision on report format and per-rule human-facing explanations. Downstream PRD.
- **CDQ-008 run-level output**: Verify that `runResult.runLevelAdvisory` findings are actually surfaced in the user-facing run report. CDQ-008's output goes to this field at coordinator level; if nothing in the output layer displays it, findings are silently dropped.
- **Rule documentation update (post-audit)**: After the audit is complete, directly edit rule documentation in this repo (`docs/rules-reference.md`) and in the eval repo (`docs/research/eval-target-criteria.md` at `~/Documents/Repositories/spinybacked-orbweaver-eval`) to reflect audit decisions (deletions, registrations, promotions, message changes) and to add each rule's OTel spec relationship — directly spec-grounded with citation, indirectly consistent, or project-specific concern not mandated by the spec (Decision 6). This work is done in M7, not deferred to a separate issue.
- **COV-005 receiver filter fragility**: `extractSetAttributeName` in `src/languages/javascript/rules/cov005.ts` filters span receivers by name pattern (`/span|activeSpan|parentSpan|rootSpan|childSpan/i`). Span variables named outside this set (e.g., `s`, `op`) have their `setAttribute` calls silently missed, producing false positives. Does not affect agent output in practice (agent consistently uses `span`), but latent bug for user-modified code. Accepted as-is for this audit; track as a future quality improvement.

## CDQ Rules

Covers: CDQ-006 (advisory), CDQ-007 / CDQ-009 / CDQ-010 (orphaned), CDQ-008 (cross-file rule).

| Rule ID | What it detects | Detection logic | Agent fix when fired | Safety | Decision | Rationale |
|---------|----------------|-----------------|----------------------|--------|----------|-----------|
| CDQ-006 (expensive attribute computation guarded) | `setAttribute` calls on span receivers where the value argument contains an expensive computation (`.map()`, `.reduce()`, `.filter()`, `.join()`, `.flatMap()`, `JSON.stringify()`) without an `isRecording()` guard — either an enclosing `if (span.isRecording())` or an early-return `if (!span.isRecording()) return` | sound | Wrap the expensive `setAttribute` call in an `if (span.isRecording())` guard | safe | keep (advisory) | Detection uses ts-morph AST parsing; trivial conversions (`.toISOString()`, `String()`, etc.) are correctly exempted. Fix is clearly described and safe — the guard preserves full telemetry when sampling is on and skips computation when not recording. A missing guard is a performance concern, not a correctness violation, so advisory status is appropriate. |
| CDQ-007 (attribute data quality — PII names, filesystem paths, nullable access) | Three sub-checks on `setAttribute` calls: (1) key is a known PII name (e.g., `author`, `email`, `username`); (2) value identifier name contains path/dir/file tokens; (3) value is `obj.property` non-optional member access without a null guard in scope. Receiver filter requires `/span/i` in the variable name — more conservative than CDQ-006. | sound | (1) Remove the attribute or rename key to a non-identifying name; (2) replace path variable with `path.basename()` or a relative equivalent; (3) add `if (obj)` guard or use optional chaining `obj?.property` | safe | refactor | Detection is AST-based for all three sub-checks. PII check and nullable check are sound. Path sub-check requires refactor (Decision 7): currently flags all path-like values uniformly, producing false positives on spec-correct `file.path` usage. Split into two: (1) privacy/security check — flag path-like values when attribute key is NOT an OTel `file.*` semantic convention key; (2) OTel exemption — when key IS `file.*`, full path is correct per the `file.path` semantic convention, do not flag. PII message was fixed (see Action Items). Was orphaned — registered in this audit. |
| CDQ-010 (untyped string method on property access) | `setAttribute` value arguments where a string-only method (`.split()`, `.slice()`, `.trim()`, `.replace()`, `.toLowerCase()`, etc. — 20 total) is called directly on a property access expression (`obj.field`) without type coercion. Checks the value argument and all descendants. Deduplicates findings by line. Does NOT fire on `String(obj.field).method()`, simple identifier receivers, call-expression receivers, or string literals. `/span/i` receiver filter. | sound | Wrap the property access in `String(obj.field)` coercion, or for timestamps use `new Date(obj.field).toISOString()` | safe | keep (advisory) | Detection is AST-based and genuinely narrow — only fires on the unsafe pattern, not on already-safe alternatives. Fix is mechanical and safe. Advisory rather than blocking because CDQ-010 can produce false positives: if the application guarantees the field is a string, the code is correct and should not fail. Was orphaned alongside CDQ-007 and CDQ-009 — registered in this audit. Agent prompt updated with preventive guidance. |
| CDQ-009 (undefined guard on span attribute values) | `setAttribute` calls where the value is a non-optional property access (`obj.prop`) and the enclosing `if` guard uses `!== undefined` (strict inequality) rather than `!= null` or a truthy check. Only fires when the `setAttribute` is inside the guarded then-branch. Handles `&&` compounds. Uses `/span/i` receiver filter. | sound | Replace `!== undefined` with `!= null` in the guard condition | safe | keep (advisory) | Detection is AST-based and narrow — only the specific pattern the agent produces when imperfectly following CDQ-007's nullable access guidance. Fix is a one-operator change, mechanically precise and safe (strictly more protective). Was orphaned alongside CDQ-007 — registered in this audit. Agent prompt updated with preventive guidance: "Do NOT use `!== undefined` as a null guard before property access." |
| CDQ-008 (consistent tracer naming convention — cross-file rule) | Post-run cross-file check: extracts all `trace.getTracer("name")` calls across instrumented files via regex, classifies names into pattern categories (dotted-path, module-name, other), flags if more than one category appears in a single run. Lives in `src/validation/tier2/cdq008.ts`; runs at coordinator level; results pushed to `runLevelAdvisory`, not per-file feedback. The per-file `cdq008Rule` in `JS_RULES` always passes — it is a registry placeholder only. | sound | Agent does not see or act on CDQ-008 findings — findings go to `runLevelAdvisory` at coordinator level, outside the per-file fix loop. No directed agent action occurs. | N/A | delete | The detection-without-fix architecture is the fundamental problem. CDQ-008 fires after all files are instrumented; there is no per-file mechanism to correct inconsistency the agent cannot see within a single file. The user-facing result is: spiny-orb produces inconsistent tracer names and tells the user to fix them — something the user did not cause. The correct solution is prevention: inject a canonical tracer name into every per-file prompt before instrumentation starts, then verify with a per-file blocking check. This makes post-hoc detection redundant. In practice the agent already produces consistent names from context (all 12 files in the commit-story-v2 instrumented branch use `'commit-story'`), confirming the approach is sound. See PRD #505. |

### OTel Spec Alignment — CDQ Rules (retroactive, Decision 4, 2026-04-18)

Starting from M2, every rule audit includes an OTel spec check (step 2 of the methodology). CDQ rules were audited before this requirement was established and are assessed here retroactively.

| Rule | OTel spec alignment | Notes |
|------|---------------------|-------|
| CDQ-006 | **Aligned** | OTel API spec explicitly endorses `isRecording()` as the optimization gate: "A Span SHOULD avoid doing expensive computations when it's not recording." ([IsRecording](https://opentelemetry.io/docs/specs/otel/trace/api/#isrecording)). CDQ-006 enforces a stated OTel best practice. |
| CDQ-007 (PII sub-check) | **Indirectly consistent** | OTel spec does not prohibit PII attribute keys by name, but data minimization aligns with the spirit of OTel conventions. Project-specific in implementation. |
| CDQ-007 (path sub-check) | **Partially aligned — refactor required** | The OTel `file.path` semantic convention ([file attributes](https://opentelemetry.io/docs/specs/semconv/attributes-registry/file/)) uses full paths; the current check flags all path-like values uniformly, producing false positives on spec-correct `file.path` usage. Refactored per Decision 7: exempt `file.*` keys, flag non-`file.*` keys as a privacy/security concern. |
| CDQ-007 (nullable access sub-check) | **Aligned** | OTel attribute values must be non-null primitives ([OTel Common — Attributes](https://opentelemetry.io/docs/specs/otel/common/#attribute)). Preventing `undefined` from reaching `setAttribute` enforces the spec's type contract. |
| CDQ-009 | **Aligned** | Same basis as CDQ-007 nullable sub-check. `!= null` (catches both null and undefined) is strictly more protective than `!== undefined` against invalid attribute values. |
| CDQ-010 | **Aligned** | OTel string attributes must be strings. Calling string-only methods on an uncoerced property access risks passing a non-string to `setAttribute`, violating the attribute value type contract. |
| CDQ-008 (deleted) | **Deletion aligned** | OTel spec defines tracer scope name as the instrumentation library name ([TracerProvider](https://opentelemetry.io/docs/specs/otel/trace/api/#tracerprovider)) but mandates no naming convention. Post-hoc inconsistency detection without a fix mechanism is not OTel-motivated. Canonical name injection (PRD #505) is more spec-aligned. |

## COV Rules

Covers: COV-004 (async operations have spans), COV-005 (domain-specific attributes present) — both advisory. COV-003 (`isExpectedConditionCatch` exemption) reviewed as Decision 5; working as intended per OTel Recording Errors spec, no changes made.

| Rule ID | What it detects | Detection logic | Agent fix when fired | Safety | Decision | Rationale |
|---------|----------------|-----------------|----------------------|--------|----------|-----------|
| COV-004 (async operations have spans) | Async functions (function declarations, variable-assigned arrow functions, class methods, and CJS exported functions) that have `async` keyword or `await` expressions but contain no `.startActiveSpan` or `.startSpan` call in their body. Pure sync functions are never flagged. Two message variants: exported functions receive a stronger directive that explicitly dismisses context propagation as an exemption and names RST-004 inapplicability; non-exported functions receive a standard directive. | sound | Add a span wrapping the function's body. Exported: "Add a span wrapping this function's body. Context propagation is not a valid exemption for exported async functions. RST-004 (unexported function) does not apply here — this function is exported. The only RST exemption that applies to exported async functions is RST-001 (utility function with no meaningful I/O or latency to track)." Non-exported: "Async functions and await expressions require spans for latency tracking and error visibility. Add a span wrapping this function's body." | safe | keep (advisory) | Detection uses ts-morph AST throughout. `hasDirectAwait` correctly stops at nested function scope boundaries, preventing a sync outer function from being flagged due to an async inner callback. `hasSpanCall` string search on the full body text is intentional — any span anywhere in the body (including nested callbacks) clears the outer function, avoiding double-flagging. Advisory rather than blocking because some async functions are legitimately span-free (RST-001 utility functions). Message strengthened in this audit: removed "Consider" language, directive wording added, RST-001 exemption clarified. OTel-spec-grounded. |
| COV-005 (domain-specific attributes present) | For each span whose name matches a registry definition, checks that all required and recommended attributes from that definition are present as `setAttribute` calls within the span's scope. No-ops when the registry is empty. Handles both `startActiveSpan` (searches callback body) and `startSpan` (walks sibling statements until `spanVar.end()`). | sound | "Required (must add): [list]. Recommended (should add): [list]. Add setAttribute() calls for each listed attribute." Agent infers values from surrounding function code. | safe | keep (advisory) | Detection is AST-based using ts-morph. Receiver filter (`/span|activeSpan|parentSpan|rootSpan|childSpan/i`) correctly handles agent-generated code but is fragile for user-modified code with non-standard span variable names — accepted as-is (see Cross-cutting concerns). Advisory rather than blocking: missing recommended attributes don't gate correctness; missing required attributes are a coverage gap but not a structural failure. Dead code (`escapeForRegExp`) removed and message tightened in this audit: required/recommended distinction now inline with attribute lists; "improve trace quality" replaced with type-aware labels. Registry-grounded (project-specific in inputs, OTel-consistent in approach). |

### OTel Spec Alignment — COV Rules

| Rule | OTel spec alignment | Notes |
|------|---------------------|-------|
| COV-003 (reviewed, not advisory) | **Directly aligned** | OTel Recording Errors spec: "Errors that were retried or handled (allowing an operation to complete gracefully) SHOULD NOT be recorded on spans." ([Recording errors](https://opentelemetry.io/docs/specs/semconv/general/recording-errors/)). The `isExpectedConditionCatch` exemption (no rethrow = expected condition = no error recording) is correct per spec. See Decision 5. |

COV-005 enforces the same pattern via the project's Weaver registry rather than the public semconv registry. The concept is OTel-aligned; the specific attributes are project-defined. |

## NDS Rules

Covers: NDS-004 (exported function signature preservation), NDS-005 (control flow preservation — includes NDS-005b sub-check assessment), NDS-006 (module system preservation) — all advisory before this audit.

| Rule ID | What it detects | Detection logic | Agent fix when fired | Safety | Decision | Rationale |
|---------|----------------|-----------------|----------------------|--------|----------|-----------|
| NDS-004 (exported function signature preservation) | Parameters added, removed, or renamed on any exported function in the instrumented output; or an exported function removed entirely. Covers all JS export forms: ESM named/default, `export const foo = () =>`, `exports.foo = function()`, `module.exports.foo`, `module.exports = { foo }`, `export default` arrow/function expressions. Uses ts-morph AST; compares parameter name lists between original and instrumented source. | sound | Instrumentation must not add, remove, or rename parameters on exported functions; all exported functions must be preserved with their original signatures. | safe | promote to blocking | OTel instrumentation never requires changing an exported function's parameter list. The advisory comment ("may produce false positives when instrumentation legitimately restructures code") names no concrete false-positive path; parameter list comparison is a direct AST equality check. When the rule fires, the agent has done something wrong, and the fix is always correct. Promoted to blocking. |
| NDS-005 (control flow preservation) | Catch clauses removed from existing try/catch blocks; throw statements modified (added or removed) inside catch blocks; finally clauses removed; entire try/catch/finally blocks missing from instrumented output. OTel-only try/finally wrappers (span lifecycle) are correctly excluded. Uses ts-morph structural fingerprinting: body anchor (first non-OTel statement in try body) with catch-clause fallback for OTel-wrapped bodies. | sound | Instrumentation must preserve existing error handling structure — do not remove or merge try/catch/finally blocks, remove catch clauses, or modify throw behavior. | safe | promote to blocking; remove LLM judge | Judge was a safety net for unknown false-positive paths, not any identified case. The anchor + catch-clause fallback matching already handles the main OTel instrumentation pattern (startActiveSpan wrapping the try body). Removing the judge makes the rule deterministic and synchronous. Promoting to blocking: the matching algorithm handles known OTel patterns; no legitimate instrumentation removes catch clauses or modifies throw behavior. |
| NDS-006 (module system preservation) | Instrumented output introduces the wrong module system — ESM `import`/`export` statements in a CJS file, or `require()`/`module.exports` assignments in an ESM file. Identifies new patterns by comparing against original signal set. Mixed-signal originals (pass — can't enforce) and unknown originals (no signals to compare — pass) are handled explicitly. Uses ts-morph AST throughout. | sound | Use the same module system as the original file: `import`/`export` for ESM, `require()`/`module.exports` for CJS. | safe | promote to blocking | Detection is sound and edge cases are explicitly handled. The advisory status was inherited from the PRD #135 batch default rather than a documented decision. No realistic false-positive path: the rule passes when it cannot enforce (mixed/unknown originals) and fires only when a clear module system mismatch is introduced. Fix is always correct when fired. Promoted to blocking. |
| NDS-007 (expected-condition catch blocks must not gain error recording) — **new rule, replaces NDS-005b** | `recordException()` added to a catch block whose corresponding original catch was an expected-condition catch (no rethrow per `isExpectedConditionCatch()` — returns a default value, returns empty, handles via `continue`, or has an empty body). Matched by body anchor to compare instrumented catch against original catch. | sound | Remove `recordException()` and `setStatus(ERROR)` from this catch block — it handles a graceful failure, not an error the caller sees. | safe | implement as blocking production rule | NDS-005b was documented and had eval coverage but no production enforcement. The agent could over-record errors in expected-condition catches and the fix loop would not catch it. Implemented as NDS-007 using `isExpectedConditionCatch()` from cov003.ts for consistent heuristics. Directly OTel-spec-grounded. |

### OTel Spec Alignment — NDS Rules

| Rule | OTel spec alignment | Notes |
|------|---------------------|-------|
| NDS-004 | **Project-specific concern — not OTel-mandated** | OTel spec does not specify function signatures. Preserving signatures is a non-destructive instrumentation constraint: the agent must not break callers of instrumented code. Indirectly consistent with the principle that instrumentation must not alter application semantics. |
| NDS-005 | **Project-specific concern — not OTel-mandated** | OTel spec does not require preserving existing error handling structure. Non-destructive instrumentation constraint: the agent must not destroy application error handling. Indirectly consistent with the OTel goal of transparent instrumentation that does not alter application semantics. |
| NDS-006 | **Project-specific concern — not OTel-mandated** | OTel spec does not dictate module systems. Module system mismatches break the target project's build. Non-destructive instrumentation constraint: the agent must not introduce incompatible import/export syntax. |
| NDS-007 | **Directly aligned** | OTel Recording Errors spec: "Errors that were retried or handled (allowing an operation to complete gracefully) SHOULD NOT be recorded on spans." ([Recording errors](https://opentelemetry.io/docs/specs/semconv/general/recording-errors/)). Expected-condition catches handle errors gracefully (caller sees success); recording them as span errors misrepresents operation outcome. Same spec basis as COV-003 Decision 5. |

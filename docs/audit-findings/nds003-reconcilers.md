## Summary

### Gap count

3 reconciler patterns have no dedicated test coverage:

1. `normalizeLine` arrow function paren strip (`(x) =>` → `x =>`) — no dedicated test
2. `reconcileReturnCaptures` multi-line object literal sub-case (`return {` → `const var = {`) — no dedicated test; the comment at line 231 confirms this path was added defensively when the prompt discouraged the pattern
3. `isSpanCallbackComma` inline filter — tested as part of a larger scenario (nds003.test.ts:2178) but not isolated; a regression would be silent until a broader test failed

### Order-dependency assessment

Execution order matters. Three dependencies are structural (not obvious from function signatures alone):

**Dependency 1 (critical): `reconcileObjectLiteralExpansion` must run before `reconcileAgentSplitLines`.**
reconcileObjectLiteralExpansion try-c handles the "incomplete group / consumed closing" case (`});`, `})`, `}` appended and retried). The comment at lines 439–443 in reconcileAgentSplitLines explicitly states that a symmetric "consumed-closing recovery" was considered for reconcileAgentSplitLines and rejected: it matched multi-line object literals like `{ includeEmpty: false, timeout: 3000 }` and caused `nds003.test.ts:1484` to silently pass by falsely reconciling a real change. If the call order is reversed, reconcileAgentSplitLines would encounter lines that reconcileObjectLiteralExpansion try-c has not yet consumed, potentially missing legitimate matches or triggering the rejected false-positive pattern. This order dependency is invisible from the call site and documented only in a comment.

**Dependency 2 (significant): `reconcileIndentReformat` must run after `reconcileObjectLiteralExpansion` and `reconcileAgentSplitLines`.**
reconcileIndentReformat uses a 50% prefix-length threshold for fuzzy group matching. At the point it runs, reconcilers 1 and 2 have already resolved the simpler exact-match cases and some intermediate Prettier-expansion cases. If reconcileIndentReformat ran first, its prefix match would encounter larger groups and could over-match, consuming lines that reconcileObjectLiteralExpansion or reconcileAgentSplitLines would have matched more precisely. The comment at lines 575–581 acknowledges a "known limitation" when two adjacent re-split statements occupy consecutive added lines — this limitation is partially mitigated by reconcilers 1 and 2 reducing the addedLines set before reconcileIndentReformat processes it.

**Dependency 3 (soft): `reconcileReturnCaptures` after `reconcileAgentSplitLines`.**
If the return expression was split by the agent onto two addedLines (e.g., `const result =\n  computeExpression(args);`), reconcileAgentSplitLines joins the two lines first. After that join, reconcileReturnCaptures sees a single `const result = computeExpression(args)` entry that matches the original `return computeExpression(args)`. If the order were reversed, reconcileReturnCaptures would not find the split form in its capture index and would miss the reconciliation. This dependency is implicit — it works because of array mutation order, not because of any code structure that makes it explicit.

Reconcilers 7–9 (`reconcileSetAttributeCaptures`, `reconcileSetAttributeMultilineArgs`, `reconcileStartActiveSpanMultilineArgs`) use `allInstrumentedLines` for context lookup (the full unfiltered output, which does not change). Their relative order is unimportant. They must run after the `rawTrimmedInstrumentedLines` variable is computed (line 1119), which is a code structure constraint, not a semantic one.

### PRD #845 M1 design assessment

**Recommendation: Proceed with PRD #845 M1, but revise its scope.**

The content-aware classifier approach in PRD #845 M1 is directionally correct — the accumulation of 13 reconciler patterns (9 named functions + 3 normalizeLine sub-cases + 1 inline filter) demonstrates the diff-based approach is straining. However, the patterns fall into two structurally different groups, and PRD #845 M1 should target only one of them.

**Group A — Prettier formatting artifacts (4 reconcilers):** reconcileObjectLiteralExpansion, reconcileAgentSplitLines, reconcileIndentReformat, reconcilePartialArgument. All four exist because Prettier reformats code differently at different indentation depths, and the agent's output at a deeper indent level has lines formatted differently than Prettier formatted them in the original. These are the most complex reconcilers (4 tries in reconcileObjectLiteralExpansion, 3 in reconcileAgentSplitLines, prefix matching with a magic-number threshold in reconcileIndentReformat). A content-aware classifier must also handle this, because the agent's output will still be at a deeper indent. The better approach is not a classifier but a normalization change: apply Prettier to both sides at the same indentation depth before comparison. This would eliminate all four Group A reconcilers.

**Group B — Semantic instrumentation patterns (5 named reconcilers + 5 normalizeLine sub-cases + 1 inline filter):** reconcileReturnCaptures, reconcileMethodChainCollapse, reconcileSetAttributeCaptures, reconcileSetAttributeMultilineArgs, reconcileStartActiveSpanMultilineArgs, plus normalizeLine's catch binding / as-const / braceless-if / arrow-paren-strip / preamble-comment-strip sub-cases and the isSpanCallbackComma filter. Two sub-cases warrant a note:

- **arrow-paren-strip**: Suppresses false positives when the agent adds parens to single-parameter arrow callbacks during span reformatting (e.g., `x =>` becomes `(x) =>`). This is agent instrumentation behavior, not a Prettier artifact — it belongs in Group B.
- **preamble-comment-strip**: Handles `// Imports used by this function` and similar buildContext annotations that the context builder injects into the agent's input but the agent never preserves in output. This is a context system artifact (neither Prettier nor instrumentation). It falls in Group B for classification purposes but is not instrumentation-motivated — it is required as long as the context builder injects these annotations, regardless of PRD #845's approach.

These reconcilers are sound, have clear semantics, and should be kept regardless of implementation approach.

**Revised PRD #845 M1 scope:** Design an approach to eliminate Group A reconcilers — the Prettier-formatting artifacts — specifically. (The original assessment counted 13 items in A+B; the corrected count is 15, with 4 in Group A and 11 in Group B. The recommendation is unchanged.) A promising approach: after the agent instruments and before NDS-003 comparison, apply Prettier to the instrumented output at the indentation depth it was formatted at (already done via `prettierNormalize`), and also apply Prettier to the original at the same indent level. Currently, `prettierNormalize` normalizes only the original. Normalizing both sides through the same Prettier pass would make reconcileObjectLiteralExpansion, reconcileAgentSplitLines, and reconcileIndentReformat redundant in the normalized path. The `checkNonInstrumentationDiffNormalized` call already normalizes the original — normalizing the instrumented output too is the remaining step.

PRD #845 M1 should not redesign Group B reconcilers. They handle semantic patterns that any implementation must understand.

### Issue #855 classification

**Verdict: targeting-logic**

`getCommitData` is the primary exported async orchestrator of git-collector.js — it is exported, it is async, it calls three functions in parallel, and it assembles the result. By any reasonable definition of COV-001 entry-point, it should be instrumented. The fact that it has been missed across 8+ consecutive eval runs is not consistent with a rule-interpretation issue: rule-interpretation errors tend to vary run-to-run as the LLM interprets ambiguous guidance differently each time. Consistent misses across 8+ runs indicate a structural bias in the function selection logic, not interpretation drift.

The most likely mechanism: the function-level fallback in `instrument-with-retry.ts` selects which function to instrument when file-level fails. If its selection criterion ranks functions by some structural property (e.g., "function with most internal logic", "shortest function", or "function with an existing span in a sibling file") rather than "exported primary orchestrator", `getPreviousCommitTime` may consistently win even though `getCommitData` is the correct target. This is a targeting-logic issue in `instrument-with-retry.ts`.

COV-001 vs RST-006 prompt ambiguity (Problem 2 of this PRD) affects which functions the agent decides to instrument during file-level attempts. But if file-level repeatedly fails and falls back to function-level, the fallback selector — not the prompt — determines the outcome. Eight consecutive runs of the same miss is the signature of a deterministic selector bug, not an LLM interpretation issue.
